05 — CRYPTO INTERNALS

Key Trust — msDS-KeyCredentialLink

โครงสร้าง binary blob และกระบวนการที่ DC ใช้ validate โดยไม่ต้องใช้ certificate

5.1 KeyCredential Binary Blob (TLV Format)

// msDS-KeyCredentialLink format: B:<length>:<hex_blob>:<DN>
KeyCredential {
    Version:         0x00000200,   // DWORD LE, Version 2

    // Tag 0x01 — Key Material
    KeyMaterial:     <raw public key bytes>,
    // RSA: BCRYPT_RSAKEY_BLOB + exponent + modulus
    // ECC: BCRYPT_ECCKEY_BLOB + X(32) + Y(32) bytes

    // Tag 0x02 — Key ID (DC ใช้ match กับ PKINIT request)
    Identifier:      SHA256(KeyMaterial),

    // Tag 0x04 — Key Usage
    KeyUsage:        0x01,         // 0x01=NGC, 0x07=FIDO2

    // Tag 0x06 — Creation Time (FILETIME)
    KeyCreationTime: <8 bytes FILETIME>,

    // Tag 0x08 — Device ID
    DeviceId:        <GUID 16 bytes>,

    // Tag 0x0B — Key Source
    KeySource:       0x00,         // 0x00=AD, 0x01=AzureAD
}
KeyID = SHA256( BCRYPT_KEY_BLOB_bytes )

5.2 DC Validation Flow

1
Client ส่ง PKINIT AS-REQ

PA-PK-AS-REQ เซ็นด้วย WHfB private key พร้อม KeyID ใน subjectKeyIdentifier

2
DC ค้นหา user จาก UPN

ใช้ UPN ใน PKINIT request ค้นหา user object ใน AD

3
อ่าน msDS-KeyCredentialLink ทุก entry

DC อ่าน attribute ทุก entry จาก user object

4
Match KeyID

คำนวณ SHA256(KeyMaterial) ของแต่ละ entry → match กับ KeyID ใน request

5
Verify Signature → ออก TGT

verify PKINIT signature ด้วย public key ที่ match → ออก Kerberos TGT
