12 — ADVANCED

Hardware Lifecycle

TPM Clear, เปลี่ยนเมนบอร์ด และการ cleanup stale keys ใน AD / Entra ID

WHfB private key มี fixedTPM=1 — ผูกกับ Storage Root Key (SRK) ของ TPM นั้น เมื่อ TPM ถูก clear หรือ chip เสียหาย SRK หาย → ทุก key ที่ลูกหลาน SRK หายตามทันที ไม่มีกลไก backup หรือ export

// WHfB Key Hierarchy ใน TPM
TPM Storage Root Key (SRK)          ← Hardware-fused, non-exportable
  └─ Owner Hierarchy
       └─ NGC Container (per user)
            └─ WHfB private key   ← fixedTPM=1, fixedParent=1

// key หายเมื่อ: เปลี่ยน motherboard, Clear TPM, Reimaging

Procedure เมื่อเปลี่ยนเมนบอร์ด

1
Note NgcKeyId เดิม

บันทึกจาก dsregcmd /status → NgcKeyId สำหรับ manual cleanup ทีหลัง

2
Hybrid AAD Rejoin (ถ้าจำเป็น)

ตรวจ dsregcmd /status → ถ้า AzureAdJoined=NO ต้อง rejoin ก่อน

3
User Sign-In → Provisioning ใหม่

sign-in ด้วย password ครั้งเดียว → WHfB wizard → MFA → key pair ใหม่

4
Cleanup Orphaned Key

key เก่าใน msDS-KeyCredentialLink กลายเป็น orphan → ลบออกเพื่อ hygiene

Cleanup Commands

// DSInternals module
Install-Module DSInternals -Force
Get-ADKeyCredential -Identity user01 | Select-Object DeviceId, CreationTime
Remove-ADKeyCredential -Identity user01 -DeviceId "{old-guid}"

// Entra ID (Microsoft Graph)
Get-MgUserAuthenticationWindowsHelloForBusinessMethod -UserId user01@corp.com
Remove-MgUserAuthenticationWindowsHelloForBusinessMethod `
    -UserId user01@corp.com -WindowsHelloForBusinessAuthenticationMethodId "{id}"
⚠ Suspend BitLocker ก่อนทำ TPM Operations

BitLocker recovery key ที่ seal ไว้กับ TPM PCR values อาจเจอปัญหาถ้าทำ TPM clear โดยไม่ suspend ก่อน
